## Detailed findings

### CR-1 — Unauthenticated POST creates attendees for arbitrary events

- **Title:** Public `POST /api/attendees` lacks auth and org/event authorization  
- **Location:** `src/pages/api/attendees.ts` (handler); `src/middleware.ts` line 142 (`pathname === '/api/attendees' && request.method === 'POST'` bypasses auth); `src/lib/db.ts` `createAttendee` (no membership check)  
- **Confidence:** High  
- **Risk:** Any client can add attendees (PII) to any event if they know or guess a valid `eventId` UUID. Rate limit is IP-only and does not establish tenant trust.  
- **Fix:** Require a signed-in user and `requireEventManage` (or at minimum `requireEventAccess` with explicit product decision) before `createAttendee`. Remove the middleware exception for this POST or replace RSVP flow with a dedicated signed/tokenized endpoint if unauthenticated RSVP is required.

```142:142:src/middleware.ts
  if (pathname === '/api/attendees' && request.method === 'POST') return next();
```

### HI-1 — Resend webhook signature not verified

- **Title:** `verifyWebhookSignature` accepts payloads without cryptographic verification  
- **Location:** `src/pages/api/webhooks/resend.ts` lines 14–17 (`return true; // TODO: Implement HMAC verification`); also `if (!secret) return true` allows unauthenticated processing when secret unset  
- **Confidence:** High  
- **Risk:** Forged webhooks can write `email_events`, flip `attendees.email_bounced` for arbitrary emails, and cause deliverability/suppression side effects.  
- **Fix:** Implement Resend’s signing scheme (or Svix if applicable) and **fail closed** when `RESEND_WEBHOOK_SECRET` is missing in production.

### HI-2 — In-memory rate limiting ineffective at scale

- **Title:** `checkRateLimit` uses a process-local `Map`  
- **Location:** `src/lib/rate-limit.ts`  
- **Confidence:** High  
- **Risk:** On Vercel, multiple instances reset limits independently; abusive traffic can exceed nominal caps. Same pattern affects ingest, check-in, Eventbrite sync, etc.  
- **Fix:** Use Redis / Vercel KV / Upstash with sliding window or token bucket for sensitive endpoints (at minimum: unauthenticated routes once CR-1 is fixed, and high-cost sync paths).

### HI-3 — Microsite ingest auth when key unset

- **Title:** `MICROSITE_WEBHOOK_KEY` empty rejects Bearer auth, but operational mistake leaves ingest closed; conversely weak or leaked key = full event/attendee write  
- **Location:** `src/pages/api/ingest/entry.ts` `getWebhookKey`, `isValidWebhookKey`  
- **Confidence:** Medium  
- **Risk:** Shared-secret design is acceptable if key is long, rotated, and required; ensure deployment fails CI/deploy if unset in production. Document rotation. (Elevated to HIGH if combined with predictable slugs / enumeration—still defense in depth.)

### ME-1 — `getEmailStatus` fail-open on DB errors

- **Title:** Returns `{ canSend: true }` when query throws  
- **Location:** `src/pages/api/webhooks/resend.ts` `getEmailStatus` catch block  
- **Confidence:** Medium  
- **Risk:** During DB outages, sending to bounced/complained addresses may continue.  
- **Fix:** Fail closed or queue with retry; surface degraded mode to callers.

### ME-2 — CSP allows `'unsafe-inline'` and `'unsafe-eval'`

- **Title:** Broad script/style allowances (Clerk-related)  
- **Location:** `vercel.json` `Content-Security-Policy`  
- **Confidence:** High (known tradeoff)  
- **Risk:** Reduces XSS blast-radius mitigation vs strict CSP.  
- **Fix:** Tighten incrementally (nonces/hashes) where Clerk allows; document as accepted risk.

### ME-3 — User access summary caching

- **Title:** `getUserAccessSummary` caches ~TTL in memory per instance  
- **Location:** `src/lib/db.ts` (`userAccessCache`)  
- **Confidence:** Medium  
- **Risk:** Role/org changes may be briefly stale for middleware decisions (`locals.isAdmin`, etc.).  
- **Fix:** Short TTL, invalidation on membership change, or move critical checks to DB on sensitive routes.

### ME-4 — Public webhook discovery GET

- **Title:** `GET /api/webhooks` exposes endpoint URLs and setup hints  
- **Location:** `src/pages/api/webhooks/index.ts`  
- **Confidence:** Low–Medium  
- **Risk:** Minor reconnaissance aid; not secret but narrows attack surface mapping.  
- **Fix:** Restrict to authenticated admin or static docs only.

### ME-5 — Demo QR codes in production when enabled

- **Title:** `ENABLE_DEMO_CODES=true` allows non-DB demo check-in responses  
- **Location:** `src/pages/api/checkin.ts`  
- **Confidence:** High  
- **Risk:** Training feature could be left on in prod.  
- **Fix:** Guard with explicit env + monitoring; default false (already the case).

### LO-1 — Security TODO in production path

- **Title:** Resend HMAC TODO in live webhook handler  
- **Location:** Same as HI-1  
- **Fix:** Track as debt until closed.

### LO-2 — Rate limit store growth

- **Title:** Cleanup runs on access; edge case of very high cardinality keys  
- **Location:** `src/lib/rate-limit.ts`

- **Fix:** Cap key space or use external store.

### LO-3 — Lockfile / package manager

## Positive patterns

- **Clerk middleware** gates most routes; JSON 401 for `/api/*` is consistent.  
- **Astro `security.checkOrigin: true`** for CSRF on custom API routes (`astro.config.mjs`).  
- **`requireEventAccess` / `requireEventManage`** centralize event scoping; `checkin.ts` uses them for QR and manual paths appropriately.  
- **Clerk welcome webhook** uses Svix verification (`src/pages/api/clerk/welcome.ts`).  
- **Microsite ingest** uses timing-safe Bearer comparison and IP rate limiting.  
- **CSV export** applies formula-injection sanitization (`src/pages/api/attendees/export.ts`).  
- **Zod** used for validation on several flows (`src/lib/validation.ts`).  
- **Vercel headers:** HSTS, X-Frame-Options DENY, CSP, nosniff, Referrer-Policy (`vercel.json`).  
- **Manual check-in + QR check-in** use atomic DB updates and audit logging (`logCheckInAttempt`).
