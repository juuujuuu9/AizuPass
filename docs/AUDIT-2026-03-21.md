## CRITICAL Findings

### CR-1. CSRF Protection Disabled Globally

**Location:** `astro.config.mjs:17`  
**Confidence:** High

```js
security: { checkOrigin: false }
```

Astro's `checkOrigin` is disabled with a comment that "Clerk handles its own CSRF." Clerk's CSRF protection only covers Clerk endpoints — all 21 custom API routes are unprotected. Any website can submit POST requests to `/api/checkin`, `/api/attendees`, `/api/send-email`, etc. using a victim's session cookie. The public RSVP endpoint (`POST /api/attendees`) is especially exposed since it requires no auth at all.

**Fix:** Re-enable `checkOrigin: true`. If Clerk conflicts arise, add Clerk routes to an exclusion list rather than disabling globally. Alternatively, add CSRF tokens to custom forms.

---

### CR-2. HTML Template Injection in Emails

**Location:** `src/lib/email.ts:94-96, 179, 186`  
**Confidence:** High

All email templates interpolate user-controlled strings (`firstName`, `lastName`, `email`, `company`, `eventName`, `organizationName`) directly into HTML without escaping:

```js
<p>Hi <strong>${attendee.firstName} ${attendee.lastName}</strong>,</p>
```

An attendee with `firstName = "<img src=x onerror=...>"` injects HTML into emails. While `<script>` tags are stripped by most email clients, injected HTML can alter layout, add phishing links, or embed tracking pixels.

**Fix:** Create an `escapeHtml()` utility and apply it to every interpolated value in email templates.

---

### CR-3. IDOR in Membership Deletion

**Location:** `src/pages/api/organizations/memberships/[id].ts:25`  
**Confidence:** High

```js
const removed = await removeOrganizationMembership(membershipId);
```

The endpoint verifies the caller owns *an* organization but does not verify `membershipId` belongs to *their* organization. An organizer of Org A could delete memberships from Org B by supplying Org B's membership IDs. UUIDs reduce practical exploitability but this is a logic flaw.

**Fix:** Change to `removeOrganizationMembership(organization.id, membershipId)` and add `WHERE organization_id = $1` to the DB query.

---

### CR-4. No Transaction Support for Multi-Statement Operations

**Location:** `src/lib/db.ts` (throughout)  
**Confidence:** High

The Neon HTTP driver does not support transactions. Critical multi-step operations run as independent queries:

- `createOrganizationForOwner`: org insert + membership insert (orphaned org on failure)
- `acceptOrganizationInvitation`: read + insert + update (double-acceptance possible)
- `deleteEventForUser`: delete attendees + delete event (attendees lost if event delete fails)

**Fix:** Switch to Neon's `transaction()` API or use the WebSocket pool mode for operations requiring atomicity.

## HIGH Findings

### HI-1. No Security Headers

**Location:** `astro.config.mjs`, no `vercel.json` headers  
**Confidence:** High

No CSP, X-Frame-Options, X-Content-Type-Options, HSTS, Referrer-Policy, or Permissions-Policy headers are configured. The app can be framed (clickjacking), and there's no script-src policy.

**Fix:** Add security headers via Vercel middleware or `vercel.json`.

---

### HI-2. In-Memory Rate Limiting Is Ineffective on Serverless

**Location:** `src/lib/rate-limit.ts`  
**Confidence:** High

The rate limiter uses a module-level `Map` that resets on every cold start and isn't shared across Vercel function instances. Under concurrent load, every request can hit a fresh Map. The code acknowledges this limitation.

**Fix:** Replace with Vercel KV, Upstash Redis, or Vercel's built-in WAF rate limiting for production.

---

### HI-3. Unbounded Query Results — No Pagination

**Location:** `src/lib/db.ts` — `getAllAttendeesForUser`, `searchAttendeesForUser`, `getAllEventsForUser`  
**Confidence:** High

Multiple DB functions return entire result sets with no LIMIT clause. For an event with 10,000 attendees, the GET `/api/attendees` endpoint serializes the full list into a single JSON response.

**Fix:** Add LIMIT/OFFSET or cursor-based pagination to list endpoints. Add a hard upper limit (e.g., 1000) as a safety net.

---

### HI-4. CSV Injection in Export

**Location:** `src/pages/api/attendees/export.ts:6-10`  
**Confidence:** High

The export CSV escapes quotes but doesn't sanitize formula-injection characters (`=`, `+`, `-`, `@`). The import path has `sanitizeCSVValue` for this purpose but export doesn't use it. Opening the CSV in Excel could execute injected formulas.

**Fix:** Apply the same `sanitizeCSVValue` prefix-stripping logic from the import module.

### HI-5. Fail-Open on Profile Check

**Location:** `src/middleware.ts:81-84`  
**Confidence:** High

```js
} catch (err) {
  console.error('[middleware] user profile sync', err);
```

If the DB is down or the `users` table doesn't exist, the middleware assumes profiles are complete and allows requests through. A database outage removes the profile-completion gate.

## Positive Security Patterns

The codebase has strong foundations worth acknowledging:

1. **Consistent `ForUser` DB scoping** — Most queries filter by userId at the data layer, preventing cross-user access.
2. **Zod validation on critical paths** — RSVP and check-in routes have schema validation.
3. **Atomic check-in** — `checkInAttendeeIfNotCheckedIn` uses a single UPDATE with `WHERE qr_used_at IS NULL` to prevent race conditions and replay attacks.
4. **QR token strength** — 128-bit cryptographic randomness with expiration and single-use enforcement.
5. **Test bypass is properly double-gated** — Requires both `NODE_ENV !== 'production'` AND `BYPASS_AUTH_FOR_TESTS=true`.
6. **Open redirect prevention** — Signout validates callback URL starts with `/` and isn't `//`.
7. **CSV formula injection on import** — `sanitizeCSVValue` strips dangerous prefixes (just needs to be applied to export too).
8. **Rate limiting on public endpoints** — Check-in (5/min), RSVP (20/min), webhook (60/min) — even if in-memory only.
